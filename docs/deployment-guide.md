# OpenEverest + provider-huawei-elb Deployment Guide

English | [中文](deployment-guide.zh-CN.md)

This beginner-friendly guide covers deploying OpenEverest from scratch on Linux and integrating the Huawei Cloud ELB Provider plugin to automatically bind Huawei Cloud ELB to database instances.

> **Note**: OpenEverest v2 is currently in Developer Preview (`v2.0.0-dev.1`). APIs may change before GA. v1 and v2 are incompatible — do not run both in the same cluster.

---

## 1. Prerequisites

| Item | Requirement |
|---|---|
| Kubernetes cluster | 1.24+ (Huawei Cloud CCE or any standard K8s) |
| kubectl | Installed and configured with cluster access |
| Helm | v3.x ([install guide](https://helm.sh/docs/intro/install/)) |
| Git | Installed (to clone the plugin repository) |
| Huawei Cloud account | ELB creation permission; obtain AK / SK / Region / Project ID |
| Container registry | `provider-huawei-elb` image pushed (this guide uses Huawei Cloud SWR) |
| Network access | The machine running commands must reach the public internet (pull charts/images) and the K8s API server |

Verify cluster connectivity:

```bash
kubectl get nodes
```

## 2. Clone the Repository & Directory Structure

### 2.1 Clone the Repository

```bash
git clone https://github.com/weimantian/provider-huawei-elb.git
cd provider-huawei-elb
```

> The repository is private. Configure GitHub access first (SSH key or Personal Access Token).

## 3. Deploy OpenEverest Platform

### 3.1 Add the Helm Repository

```bash
helm repo add openeverest https://openeverest.github.io/helm-charts/
helm repo update
```

> If the repository URL above is unavailable, check the [OpenEverest repo](https://github.com/openeverest/openeverest) for the latest chart address.

### 3.2 Install OpenEverest Core

```bash
helm install everest-core openeverest/openeverest \
  --namespace everest-system \
  --create-namespace
```

This deploys the OpenEverest Server, Operator, and dependencies into the `everest-system` namespace.

### 3.3 Verify Deployment

```bash
kubectl get pods -n everest-system
```

All pods should be `Running`. If some are not ready, wait 1-2 minutes and retry.

### 3.4 Get Admin Password (Optional)

```bash
kubectl get secret everest-accounts -n everest-system \
  -o jsonpath='{.data.users\.yaml}' | base64 --decode
```

> The secret name and field path may vary by OpenEverest version. If the command fails, consult the [OpenEverest documentation](https://github.com/openeverest/openeverest).

### 3.5 Access the UI (Optional)

```bash
kubectl port-forward svc/everest 8080:8080 -n everest-system
```

Open `http://127.0.0.1:8080` in your browser. Log in with `admin` and the password from the previous step.

> For production, expose the UI via Ingress or a LoadBalancer Service.

## 4. Deploy the provider-huawei-elb Plugin

> **Note**: All operations in this section run on your **local Linux machine** (where `kubectl` and `helm` are installed). You **do not need to enter any container**. `provider-values.yaml` is a local file that Helm reads to configure the deployment. To change configuration later, edit the local file and run `helm upgrade` — no `docker exec` or `kubectl exec` needed.

### 4.1 Image Preparation

Deploying the Provider requires a container image. There are two ways to obtain it — **choose either one**.

#### Option 1: Use the Pre-Built Multi-Arch Image (Recommended, pull directly)

The image has been built and pushed to Huawei Cloud SWR. **It supports both amd64 (x86) and arm64 (Kunpeng) architectures**. Pull it directly:

```bash
docker pull swr.cn-north-4.myhuaweicloud.com/weimantian/provider-huawei-elb:latest
```

After pulling successfully, skip to [§4.2](#42-create-huawei-cloud-credentials-secret) to continue deployment.

#### Option 2: Build the Image Locally

If you need to modify code and rebuild, or cannot access SWR, you can build in the **repository root**.

##### Method A: Standard Build (requires network, Docker auto-pulls golang base image)

```bash
# Run in the repository root, uses the standard Dockerfile
docker build -t provider-huawei-elb:latest .
```

> This uses `Dockerfile`. Docker pulls `golang:1.26` to compile Go source, then packages the runtime with `distroless`. Requires Docker Hub access.
>
> **Note**: This builds a single-architecture image for the current machine's architecture only.

##### Method B: Local Cross-Compile Build (no network needed, uses local base image)

If the machine cannot reach Docker Hub (cannot pull `golang:1.26`), cross-compile the binary first, then package with a locally available base image:

```bash
# 1. Check cluster architecture (arm64 or amd64)
kubectl get nodes -o jsonpath='{.items[0].status.nodeInfo.architecture}'

# 2. Cross-compile the Go binary (choose GOARCH based on cluster architecture)
#    For arm64 clusters (e.g. Huawei Cloud Kunpeng CCE):
CGO_ENABLED=0 GOOS=linux GOARCH=arm64 go build -a -o bin/provider cmd/provider/main.go

#    For amd64 clusters (e.g. Huawei Cloud x86 CCE):
CGO_ENABLED=0 GOOS=linux GOARCH=amd64 go build -a -o bin/provider cmd/provider/main.go

# 3. Package with Dockerfile.local (disable BuildKit to avoid SWR-incompatible manifest list)
DOCKER_BUILDKIT=0 docker build -f Dockerfile.local -t provider-huawei-elb:latest .
```

> This uses `Dockerfile.local` with the local `redis:7-alpine` image as base (includes CA certs for HTTPS to Huawei Cloud API). The binary is statically compiled with no CGO dependencies.
>
> **Note**: This builds a single-architecture image for the specified architecture only.

##### Method C: Multi-Arch Build (supports both amd64 + arm64)

To support both x86 and Kunpeng clusters with a single image (manifest list):

```bash
# 1. Cross-compile binaries for both architectures
CGO_ENABLED=0 GOOS=linux GOARCH=arm64 go build -a -o bin/provider-arm64 cmd/provider/main.go
CGO_ENABLED=0 GOOS=linux GOARCH=amd64 go build -a -o bin/provider-amd64 cmd/provider/main.go

# 2. Build arm64 image (--provenance=false avoids SWR-incompatible attestation)
docker buildx build --platform linux/arm64 --provenance=false --load \
  -t swr.cn-north-4.myhuaweicloud.com/weimantian/provider-huawei-elb:latest-arm64 \
  -f Dockerfile.multiarch .

# 3. Build amd64 image
docker buildx build --platform linux/amd64 --provenance=false --load \
  -t swr.cn-north-4.myhuaweicloud.com/weimantian/provider-huawei-elb:latest-amd64 \
  -f Dockerfile.multiarch .

# 4. Log in to SWR
docker login -u cn-north-4@<YOUR_ACCESS_KEY> -p <YOUR_LOGIN_TOKEN> swr.cn-north-4.myhuaweicloud.com

# 5. Push both single-arch images
docker push swr.cn-north-4.myhuaweicloud.com/weimantian/provider-huawei-elb:latest-arm64
docker push swr.cn-north-4.myhuaweicloud.com/weimantian/provider-huawei-elb:latest-amd64

# 6. Combine into multi-arch manifest list and push
docker manifest create \
  swr.cn-north-4.myhuaweicloud.com/weimantian/provider-huawei-elb:latest \
  swr.cn-north-4.myhuaweicloud.com/weimantian/provider-huawei-elb:latest-arm64 \
  swr.cn-north-4.myhuaweicloud.com/weimantian/provider-huawei-elb:latest-amd64
docker manifest push swr.cn-north-4.myhuaweicloud.com/weimantian/provider-huawei-elb:latest
```

> This uses `Dockerfile.multiarch`, which selects the correct binary via `TARGETARCH`.
>
> **Critical**: `--provenance=false` is required. Without it, BuildKit generates attestation metadata that SWR rejects with `Invalid image, fail to parse 'manifest.json'`.

##### After Building: Choose Based on Cluster Type

**Scenario A: Deploying to a remote CCE cluster** (CCE nodes cannot access your local Docker images, must push to SWR)

If you used Method A/B (single-arch), push manually to SWR:

```bash
# 1. Log in to Huawei Cloud SWR (get login command from Console → SWR → Login Command)
docker login -u cn-north-4@<YOUR_ACCESS_KEY> -p <YOUR_LOGIN_TOKEN> swr.cn-north-4.myhuaweicloud.com

# 2. Tag the image (replace weimantian with your SWR organization/namespace)
docker tag provider-huawei-elb:latest swr.cn-north-4.myhuaweicloud.com/weimantian/provider-huawei-elb:latest

# 3. Push
docker push swr.cn-north-4.myhuaweicloud.com/weimantian/provider-huawei-elb:latest
```

If you used Method C (multi-arch), the image was already pushed during the build process. Skip this step.

Then use the SWR image address in `provider-values.yaml`:
```yaml
image:
  repository: swr.cn-north-4.myhuaweicloud.com/weimantian/provider-huawei-elb
  tag: latest
  pullPolicy: IfNotPresent
```

**Scenario B: Local k3d/kind dev cluster** (K8s node is your local machine, use image directly, no push needed)

```bash
# k3d: import local image into cluster
k3d image import provider-huawei-elb:latest

# kind: import local image into cluster
kind load docker-image provider-huawei-elb:latest
```

Then use the local image name in `provider-values.yaml` with pullPolicy Never:
```yaml
image:
  repository: provider-huawei-elb
  tag: latest
  pullPolicy: Never          # for local dev only, never pull from registry
```

### 4.2 Create Huawei Cloud Credentials Secret

**Do not pass AK/SK via `--set` on the command line** (it leaks to shell history). Use a Kubernetes Secret:

```bash
kubectl create secret generic huawei-cloud-credentials \
  --from-literal=ak=<YOUR_AK> \
  --from-literal=sk=<YOUR_SK> \
  --from-literal=project-id=<YOUR_PROJECT_ID> \
  -n everest-system
```

### 4.3 Create SWR Pull Secret (Private Registry)

If the SWR repository is private, create a pull secret:

```bash
kubectl create secret docker-registry swr-pull-secret \
  --docker-server=swr.cn-north-4.myhuaweicloud.com \
  --docker-username=cn-north-4@<YOUR_ACCESS_KEY> \
  --docker-password=<YOUR_LOGIN_TOKEN> \
  -n everest-system
```

> If the repository is public, skip this step and remove the `imagePullSecrets` section in `provider-values.yaml` below.

### 4.4 Create provider-values.yaml

Create `provider-values.yaml` in the **repository root** (same level as `Makefile`):

```bash
cat > provider-values.yaml <<'EOF'
image:
  repository: swr.cn-north-4.myhuaweicloud.com/weimantian/provider-huawei-elb
  tag: latest
  pullPolicy: IfNotPresent

# SWR private registry pull secret (remove these two lines if repository is public)
imagePullSecrets:
  - name: swr-pull-secret

# Huawei Cloud credentials injected via environment variables
extraEnv:
  - name: HUAWEI_CLOUD_AK
    valueFrom:
      secretKeyRef:
        name: huawei-cloud-credentials
        key: ak
  - name: HUAWEI_CLOUD_SK
    valueFrom:
      secretKeyRef:
        name: huawei-cloud-credentials
        key: sk
  - name: HUAWEI_CLOUD_REGION
    value: cn-north-4
  - name: HUAWEI_CLOUD_PROJECT_ID
    valueFrom:
      secretKeyRef:
        name: huawei-cloud-credentials
        key: project-id
EOF
```

> **This is the only file you need to create manually.** It overrides the defaults in `charts/provider-huawei-elb/values.yaml` (image address + credentials).
>
> **Important**: The Provider Helm Chart does **not** provide `config.ak/sk` fields. All Huawei Cloud credentials are injected via `extraEnv` environment variables, supporting both `value` (plaintext) and `valueFrom.secretKeyRef` (Secret reference).

### 4.5 Install the Provider

> **This step starts the container automatically**: `helm install` creates a Kubernetes Deployment, and K8s pulls the image from SWR and starts the Provider container in the cluster. You do not need to manually `docker run`.
Run from the **repository root**:

```bash
helm install provider-huawei-elb ./charts/provider-huawei-elb \
  --namespace everest-system \
  --values provider-values.yaml
```

Helm deploys:
- **Deployment** (Provider Pod)
- **Provider CR** (reads spec from `charts/provider-huawei-elb/generated/provider-spec.yaml`)
- **ClusterRole / ClusterRoleBinding / ServiceAccount** (RBAC permissions)
- **Service** (HTTP port 8082 for schema queries and health checks)

### 4.6 Verify Plugin Deployment

```bash
# 1. Provider Pod status
kubectl get pods -n everest-system -l app.kubernetes.io/name=provider-huawei-elb

# 2. Provider CR created
kubectl get provider provider-huawei-elb -n everest-system

# 3. Check Provider logs (confirm credentials loaded, no errors)
kubectl logs -n everest-system -l app.kubernetes.io/name=provider-huawei-elb --tail=30
```

Pod `Running` with no `ERROR`-level log output means deployment is successful.

## 5. Create an ELB Instance

The `examples/` directory in the repo contains 3 ready-to-use example files. Modify and apply them.

### 5.1 Public ELB (with EIP, externally accessible)

Use `examples/instance-simple.yaml`:

```bash
# 1. Edit the file, replace VPC / subnet / availability zone with your actual Huawei Cloud resource IDs
vi examples/instance-simple.yaml
```

Fields to modify (`examples/instance-simple.yaml`):

```yaml
apiVersion: core.openeverest.io/v1alpha1
kind: Instance
metadata:
  name: my-elb-simple
  namespace: everest-system          # ← add namespace
spec:
  provider: provider-huawei-elb
  topology:
    type: public-elb
    config:
      bandwidthSize: 10              # bandwidth Mbit/s
      bandwidthChargeMode: traffic
  components:
    elbEngine:
      type: elb-engine
      customSpec:
        vpcId: vpc-xxxxxxxx          # ← replace with your VPC ID
        vipSubnetCidrId: subnet-xxxxxxxx  # ← replace with your subnet ID
        availabilityZoneList:
          - cn-north-4a              # ← replace with your availability zone
```

```bash
# 2. Apply
kubectl apply -f examples/instance-simple.yaml
```

### 5.2 Internal ELB (VPC-internal only)

Use `examples/instance-internal-elb.yaml`:

```bash
# 1. Edit the file, replace VPC / subnet / availability zone
vi examples/instance-internal-elb.yaml
```

File content (`examples/instance-internal-elb.yaml`):

```yaml
apiVersion: core.openeverest.io/v1alpha1
kind: Instance
metadata:
  name: my-internal-elb
  namespace: everest-system          # ← add namespace
spec:
  provider: provider-huawei-elb
  topology:
    type: internal-elb               # internal topology, no bandwidth config needed
  components:
    elbEngine:
      type: elb-engine
      customSpec:
        vpcId: vpc-xxxxxxxx          # ← replace
        vipSubnetCidrId: subnet-xxxxxxxx  # ← replace
        availabilityZoneList:
          - cn-north-4a
    elbListener:
      type: elb-listener
      customSpec:
        protocol: TCP
        port: 3306
        backendPort: 3306
```

```bash
# 2. Apply
kubectl apply -f examples/instance-internal-elb.yaml
```

### 5.3 Full Public ELB (Custom Listener)

Use `examples/instance-example.yaml` for dual-AZ + custom listener port:

```bash
vi examples/instance-example.yaml    # replace VPC / subnet / availability zone
kubectl apply -f examples/instance-example.yaml
```

> **Omitting `elbListener`** uses defaults: protocol TCP, port 3306, backend port 3306.
>
> **Protocol note**: TCP/UDP pass through directly; HTTP/HTTPS do L7 at the ELB layer, K8s Service protocol falls back to TCP.

## 6. Verify ELB and Database Access

### 6.1 Check Instance Status

```bash
kubectl get instance -n everest-system
```

Status transitions: `Provisioning` → `Ready` (or `Failed`).

```bash
# View detailed status including ELB IP and port
kubectl get instance my-elb-simple -n everest-system -o yaml
```

Look for the `status` field:

```yaml
status:
  provisioningStatus: Ready
  connectionDetails:
    host: <ELB_PUBLIC_IP>      # EIP for public ELB; private IP for internal ELB
    port: "3306"
```

### 6.2 Check the K8s Service

The Provider creates a Service named `elb-<instance-name>`:

```bash
kubectl get svc -n everest-system
```

```
NAME                    TYPE           CLUSTER-IP    EXTERNAL-IP      PORT(S)          AGE
elb-my-elb-simple       LoadBalancer   10.x.x.x      <ELB_PUBLIC_IP>  3306:3xxxx/TCP   2m
```

Confirm `EXTERNAL-IP` is assigned (EIP for public ELB, private IP for internal ELB).

Check Service annotations (confirm ELB ID binding):

```bash
kubectl get svc elb-my-elb-simple -n everest-system -o jsonpath='{.metadata.annotations}' | jq .
```

```json
{
  "kubernetes.io/elb.id": "<ELB_ID>"
}
```

### 6.3 Verify ELB Status (Huawei Cloud Console)

1. Log in to the [Huawei Cloud Console](https://console.huaweicloud.com/) → **Elastic Load Balance (ELB)**
2. Find the instance named `elb-my-elb-simple`
3. Confirm status is **Active**
4. Check the **Listeners** tab: there should be a TCP:3306 listener
5. Check the **Backend Server Group**: backend member health should be **Healthy**

### 6.4 Test Network Connectivity

**Public ELB** (test from outside the VPC):

```bash
# Get the ELB public IP
ELB_IP=$(kubectl get instance my-elb-simple -n everest-system \
  -o jsonpath='{.status.connectionDetails.host}')
echo "ELB IP: $ELB_IP"

# TCP port connectivity test
nc -zv $ELB_IP 3306
```

**Internal ELB** (test from a machine inside the VPC):

```bash
nc -zv <ELB_PRIVATE_IP> 3306
```

Expected output: `Connection to <IP> 3306 port [tcp/mysql] succeeded!`

### 6.5 Test Database Access

Assuming the backend database is MySQL (port 3306):

```bash
# Public ELB
mysql -h $ELB_IP -P 3306 -u <DB_USER> -p<DB_PASSWORD> -e "SELECT 1"

# Internal ELB (must run from within the VPC)
mysql -h <ELB_PRIVATE_IP> -P 3306 -u <DB_USER> -p<DB_PASSWORD> -e "SELECT 1"
```

PostgreSQL (port 5432):

```bash
psql -h <ELB_IP> -p 5432 -U <DB_USER> -d <DB_NAME> -c "SELECT 1"
```

Expected output: `1` or similar result confirming database access works.

### 6.6 ELB Health Check Verification

The ELB automatically health-checks backend databases. If a backend is unreachable, the ELB marks the member as unhealthy:

```bash
# Query ELB backend health via Huawei Cloud CLI
# Install: https://support.huaweicloud.com/ptrc-hcli/hcli_01_01.html
hcloud ELB ShowMemberHealth --elb_id=<ELB_ID>
```

Or in the Huawei Cloud Console → ELB instance → **Backend Server Group** → check member health status.

> **Health check not ready**: After ELB creation, health checks take 10-30 seconds to complete. Connection timeouts during this period are normal.

## 7. Troubleshooting

| Symptom | Resolution |
|---|---|
| Provider Pod `CrashLoopBackOff` | `kubectl logs -n everest-system <pod>`. Common: missing credentials / wrong region / image pull failure |
| Instance stuck in `Provisioning` | `kubectl describe instance <name> -n everest-system`. Check VPC/subnet IDs are correct |
| Instance status is `Failed` | Provider logs contain detailed errors. Common: ELB quota exhausted / no available IPs in subnet / AZ not supported |
| Service has no `EXTERNAL-IP` | Public ELB: check EIP quota. Internal ELB: check subnet CIDR conflicts. CCE controller takes 1-3 minutes |
| ELB backend member unhealthy | Confirm database Pod is running, listening on `backendPort`, security group allows ELB → database traffic |
| Database connection timeout | Confirm ELB health check passed (Section 6.6), security group allows client → ELB inbound traffic |
| `helm install` reports `connection refused` | Check kubeconfig: `kubectl cluster-info` |
| `helm install` cannot find chart | Ensure you're in the repository root and `./charts/provider-huawei-elb` exists |

## 8. Cleanup

### Delete a Single ELB Instance

```bash
kubectl delete instance my-elb-simple -n everest-system
```

The Provider automatically deletes the corresponding ELB and K8s Service.

### Uninstall the Provider

```bash
helm uninstall provider-huawei-elb -n everest-system
```

### Uninstall OpenEverest

```bash
helm uninstall everest-core -n everest-system
kubectl delete namespace everest-system
```

> **Note**: After deleting an Instance CR, the Provider deletes the Huawei Cloud ELB first, then the K8s Service. If the Provider has already been uninstalled, you must manually delete any remaining ELBs in the Huawei Cloud Console.
